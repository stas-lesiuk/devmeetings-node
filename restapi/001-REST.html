Co to jest REST?

Aktualnie nowy produkt dodawany jest przez formularz. Zamiast tego zbudujmy porządne RESTowe API. Jak to zrobić?

REST = Representational State Transfer.

Fundamenty:

  • Client-Server architecture - podział
  • Statelessness - brak "kontekstu" requestu
  • Cacheability - serwer może powiedzieć co można zcachować
  • Layered system - load balancing, proxying
  • Uniform interface
    • Resource identification in requests - zasoby identyfikowane przez URI są rozdzielone od reprezentacji
    • Resource manipulation through representation - to co klient dostaje wystarcza do modyfikacji lub usunięcia zasobu (brak dodatkowych założeń)
    • Self-descriptive messages - każda wiadomość ma wystarczająco informacji do tego aby klient mógł ją przetworzyć. Klient nie powinien "spodziewać się" jakiegoś typu.
    • Hypermedia as the engine of application state (HATEOAS) - klient RESTowy powinien mieć możliwość automatycznie odkryć API wchodząc tylko na główny endpoint (strona główna). Odpowiedzi powinny zawierać linki do innych dostępnych zasobów.
